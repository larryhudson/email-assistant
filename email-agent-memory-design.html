High-level design

Email agents with isolated memory

A single stack can host one or more email assistants. Each inbound email webhook is normalized through an email provider adapter, routed to the right assistant, retrieves only that assistant's memory, performs work, sends a reply, then records the run for future messages and admin review.

1 stack starts with you as admin and your mum as the regular user, while still allowing your own assistant later.
isolated scope assistant, user, memory, tools, and budget are resolved before any agent run.
admin view lets you inspect runs, costs, memory, failures, and limits without exposing UI to the end user.
01 — Runtime Flow
Webhook

Receive inbound

Validate the provider webhook, then normalize body text, sender, recipient, attachments, and headers.

Route

Resolve assistant

Map the inbound address to an assistant profile, owner, end user, budget, tools, and memory scope.

Recall

Load memory

After the budget gate passes, resolve the thread and fetch memory only inside that assistant scope.

Agent

Act and draft

Call tools, produce a final reply, and keep a structured trace of important actions.

Reply + learn

Send, then persist

Reply through the email provider adapter first; then store messages, traces, and extracted durable memory.

Critical path: validation, assistant resolution, budget check, thread resolution, memory recall, agent execution, and sending the reply. Heavy summarization, embedding, and memory consolidation should run after sending.

03 — Email Threading Model

User sends email

Inbound message creates a new internal thread if no existing message headers match.

Message-ID: U1

Assistant replies

Outbound message is stored and indexed. The reply includes headers that preserve the conversation chain.

Message-ID: A1

User replies

The next webhook points at the assistant message, so the resolver maps it back to the same thread.

In-Reply-To: A1
Store and index both inbound and outbound Message-ID values with assistant_id. The same raw header must never be allowed to cross assistant scopes. Each assistant should have a separate inbound address, so routing can stay explicit and inspectable.
04 — Memory Model
Session store

Thread history

Fast lookup by assistant and thread ID. Used to answer “what has happened in this conversation?”

  • full thread history available for inspection
  • inbound emails and assistant replies
  • short tool summaries
  • prompt injection can use a recent window plus an on-demand thread inspection tool
Long-term store

Durable facts

Semantic search over extracted facts, preferences, decisions, contacts, and unresolved tasks scoped by assistant.

  • embedded memory text
  • scope and importance
  • source thread links
  • permissive capture to start, then tighten policy from observed runs
Trace store

Audit trail

Structured record of what the agent did so later runs can inspect outcomes without replaying logs.

  • tool calls
  • status and errors
  • reply metadata

06 — Admin Interface

Operator dashboard

You get a web UI; your mum only uses email. The dashboard should show recent runs, replies, failures, memory writes, and cost.

  • filter by assistant, thread, date, status, and cost
  • inspect inbound email, retrieved memory, tools used, and outbound reply
  • pause an assistant or put it into approval-required mode

Human approval mode

For higher-risk actions, the assistant can draft but not send until you approve from the admin UI.

  • useful during early rollout
  • can be enabled per assistant or per tool
  • keeps the end-user experience simple
07 — Cost and Safety Controls

Budget gate before agent run

Every inbound email checks the assistant's daily and monthly budget before calling agent models or tools.

  • hard stop when limit is reached
  • fallback reply uses a cheap template, not another model call: “I’ve hit my budget limit for this month; it resets in X days.”
  • admin notification when spend crosses thresholds

Usage ledger

Record estimated and actual cost per run so the admin UI can explain where spend came from.

  • model tokens and model cost
  • tool/API calls and external cost
  • memory extraction and embedding cost
08 — Email Provider Port Decision

Define email as a port

Runtime code should depend on normalized inbound/outbound email types, not directly on Mailgun payload fields, signatures, webhooks, or send parameters.

  • Interface: verify webhook, parse inbound email, send reply, expose provider message IDs.
  • Invariant: normalized emails must preserve Message-ID, In-Reply-To, and References.
  • Error mode: provider send failures create retryable run failures visible in admin.

Mailgun adapter

The first adapter can use Mailgun for inbound webhooks and outbound replies, while hiding Mailgun-specific field names and signature verification from the assistant runtime.

  • fastest route for the first deployment
  • keeps provider-specific parsing localized
  • lets a future SES/Postmark/Graph adapter satisfy the same interface
09 — Memory Port Decision

Define memory as a port

Agent and webhook code should depend on a small memory interface, not directly on Cognee, pgvector, prompts, or a custom schema. Cognee can be one adapter behind that seam.

  • Interface: recall context, record run, curate durable memories, delete/export assistant memory.
  • Invariant: every operation receives assistant_id and cannot cross assistant scope.
  • Error mode: recall failures degrade the run; record failures enqueue retry work after the reply is sent.

Cognee adapter

Uses Cognee for graph/vector memory, session-style recall, and memory enrichment. Best when we want off-the-shelf memory behavior quickly.

  • more capability up front
  • less local memory code
  • extra dependency, operational complexity, and adapter mapping work

Minimal local adapter

Uses our own tables, embeddings, and memory extraction prompt. Best when predictability, inspectability, and cost control matter more than graph features.

  • simple to inspect and debug in the admin UI
  • direct control over retention and budgets
  • we own more retrieval and memory-curation behavior

Recommendation

Keep the port either way. For the first version, choose the adapter based on the fastest route to a trustworthy admin experience, not on theoretical memory sophistication.

  • start with one adapter, but design the seam as real because a test adapter is also needed
  • write behavior tests at the memory port, not against Cognee internals
  • avoid leaking Cognee terms into assistant runtime code
10 — Design Decisions

Send first, enrich after

The reply should not wait for summarization, embeddings, or durable memory extraction unless the same run needs the new memory immediately.

  • lower perceived latency
  • fewer user-visible failures
  • requires background job retry handling

Extract memories, do not embed everything

Raw emails stay in thread history. Long-term memory can start permissive, then tighten once real runs show what creates noise.

  • early version stores broad facts, preferences, tasks, decisions, and useful context
  • admin review can delete or demote bad memories
  • policy becomes stricter after observing actual email patterns

Own the thread resolver

The email provider handles delivery, but the app owns the durable mapping from message headers to internal thread sessions.

  • portable across providers
  • supports outbound indexing
  • needs idempotency keys for retries

Keep trace separate from memory

Traces are useful for audit and debugging, but only selected trace summaries should become long-term memory.

  • clearer operational logs
  • better privacy controls
  • easier deletion by assistant, end user, or thread
11 — Deepening Opportunities

Assistant router module

Give callers one interface: resolve a normalized inbound email into an assistant run scope. Hide recipient routing, assistant config, tool allowlists, budget IDs, and disabled/paused states.

  • Leverage: every webhook path starts with the same isolation decision.
  • Locality: multi-assistant routing bugs live in one module.
  • Test surface: normalized inbound email in, assistant scope or rejection out.

Thread resolution module

Give callers one interface: resolve a normalized inbound email and assistant scope into an internal thread. Hide provider IDs, header normalization, outbound message indexing, fallback creation, and idempotency behind it.

  • Leverage: every webhook path gets the same threading behavior.
  • Locality: reply-chain bugs live in one module instead of every handler.
  • Test surface: header fixtures in, thread decision out.

Run recorder module

Instead of manually writing inbound email, outbound reply, trace rows, and memory jobs from the handler, expose one “record completed run” interface.

  • Leverage: one call persists the full operational record.
  • Locality: retry, dedupe, and partial-write behavior sit together.
  • Test surface: completed run in, expected durable writes/jobs out.

Memory curator module

Make durable memory extraction a module with a small interface: transcript in, accepted memories out. Keep extraction prompts, filtering, importance scoring, and embedding writes inside.

  • Leverage: cleaner recall without every caller learning memory policy.
  • Locality: privacy and retention rules are concentrated.
  • Test surface: transcript cases assert accepted/rejected memories.

Reply envelope module

Build outbound subject, recipients, In-Reply-To, References, and generated Message-ID in one place.

  • Leverage: correct email threading for every outbound reply.
  • Locality: provider quirks and RFC-ish header rules do not leak into agent code.
  • Test surface: normalized inbound email in, normalized outbound reply out.

Budget governor module

Expose one decision interface for cost control: given an assistant and planned run, allow, degrade, require approval, or send the budget-limit reply.

  • Leverage: all model/tool calls share the same spend policy.
  • Locality: cost limits, reset dates, thresholds, and fallback copy sit together.
  • Test surface: ledger state plus run estimate in, decision out.
12 — Review Questions

Decisions captured

  • You are the admin/operator; your mum is the regular end user.
  • Each assistant gets a separate inbound email address.
  • Budget limit behavior is an explanatory reply with days until reset.
  • The agent can inspect the full thread and memory; prompt injection should start with a recent window plus on-demand inspection tools.
  • Durable memory capture can be permissive for the first version.
  • Email provider should sit behind a port; Mailgun is the first adapter.
  • Memory should sit behind a port so Cognee and a local adapter can be swapped.
